1. Introdução

Os sistemas de gerenciamento de banco de dados (SGBDs) ocupam posição central na infraestrutura tecnológica contemporânea. Desde os sistemas transacionais que processam operações financeiras até as plataformas de atendimento ao cliente, passando pelos aplicativos móveis que operam em condições de conectividade intermitente, a gestão eficiente e confiável dos dados constitui requisito indispensável para a continuidade dos negócios e a integridade das informações.

O presente trabalho tem por objetivo explorar os fundamentos teóricos e práticos dos bancos de dados relacionais e não relacionais, com ênfase nos conceitos de arquitetura de SGBD, propriedades ACID, controle de concorrência, recuperação de falhas, segurança e integridade, bancos de dados distribuídos, bancos móveis, paradigma NoSQL e MapReduce. A abordagem adotada integra a revisão da literatura especializada com a implementação de um protótipo funcional denominado SmartTicket DB.

O SmartTicket DB é um sistema acadêmico de gestão de chamados de suporte desenvolvido em Python, utilizando SQLite como banco de dados relacional e arquivos JSON como simulação de um banco de dados NoSQL documental. A escolha por essas tecnologias foi deliberada: o SQLite é amplamente adotado em protótipos acadêmicos e em ambientes de desenvolvimento por dispensar configuração de servidor, enquanto os arquivos JSON reproduzem fielmente a estrutura de documentos armazenados em sistemas como MongoDB, Firebase ou Firestore. O protótipo não tem a pretensão de ser um produto final, mas de demonstrar, de forma clara e executável, como os conceitos teóricos se materializam em código.

A justificativa para a adoção de uma arquitetura híbrida SQL/NoSQL reside na complementaridade entre os dois paradigmas: os dados transacionais, que exigem consistência rígida e integridade referencial, são melhor servidos por um modelo relacional com propriedades ACID; os dados de log, eventos e histórico de alterações, que possuem estrutura variável e alto volume de escrita, são mais eficientemente armazenados em um modelo documental flexível. Esta combinação é observada em sistemas reais de grande escala, como os aplicativos de delivery e atendimento analisados no Capítulo 3.

2. Fundamentação Teórica

2.1 Arquitetura de SGBD e Independência de Dados

A arquitetura de três esquemas proposta pelo comitê ANSI/SPARC divide a estrutura de um sistema de banco de dados em três níveis: interno, conceitual e externo. O nível interno, também denominado nível físico, descreve como os dados são efetivamente armazenados nos dispositivos de hardware, incluindo estruturas de arquivo, índices e alocação de blocos de disco. O nível conceitual define a estrutura lógica global do banco, com as entidades, atributos, relacionamentos e restrições de integridade. O nível externo, por sua vez, apresenta visões personalizadas dos dados para diferentes grupos de usuários ou aplicações, exibindo apenas as informações relevantes para cada contexto (ELMASRI; NAVATHE, 2011).

O objetivo central desta arquitetura é, conforme sintetizado por Elmasri e Navathe (2011), "separar as aplicações do usuário do banco de dados físico", garantindo que modificações no nível de armazenamento físico não exijam alterações nos programas de aplicação (independência de dados física) e que mudanças no esquema conceitual global não afetem os esquemas externos (independência de dados lógica).

No contexto do protótipo SmartTicket DB, esta arquitetura se manifesta da seguinte forma: a aplicação Python (nível externo) interage com as tabelas usuarios, chamados e logs_sql (nível conceitual) por meio de funções abstratas, sem precisar conhecer os detalhes de como o SQLite organiza os dados no arquivo .db no disco (nível interno). A substituição do SQLite por PostgreSQL, por exemplo, exigiria apenas a alteração da camada de conexão, sem impacto na lógica de negócio.

2.2 Transações e Propriedades ACID

Uma transação é definida como uma unidade lógica indivisível de trabalho, composta por uma ou mais operações de acesso ao banco de dados — leituras e escritas. Uma transação deve ser executada em sua totalidade ou, em caso de falha, desfeita completamente, de modo que o banco retorne ao estado anterior ao início da operação (ELMASRI; NAVATHE, 2011).

As propriedades que garantem a confiabilidade das transações são agrupadas sob o acrônimo ACID, conforme definido por Silberschatz, Korth e Sudarshan (2020):

  • Atomicidade: todas as operações de uma transação são executadas ou nenhuma é. O princípio "tudo ou nada" garante que não existam estados parcialmente modificados no banco. Em caso de falha, o mecanismo de rollback desfaz todas as alterações realizadas até aquele ponto.
  • Consistência: uma transação leva o banco de dados de um estado válido e consistente a outro estado igualmente válido. As restrições de integridade (chaves primárias, estrangeiras, domínios) devem ser satisfeitas ao final de cada transação.
  • Isolamento: transações executadas simultaneamente não interferem umas nas outras. O resultado de transações concorrentes deve ser equivalente ao resultado de sua execução serial. O nível de isolamento pode ser configurado em SGBDs modernos para equilibrar desempenho e consistência.
  • Durabilidade: após a confirmação de uma transação (COMMIT), as mudanças realizadas persistem definitivamente no banco de dados, mesmo diante de falhas de hardware ou software subsequentes.

Os comandos que delimitam uma transação são: BEGIN (ou implícito no SQLite), que marca o início; COMMIT, que confirma todas as operações; e ROLLBACK, que as desfaz. No protótipo, a abertura de um chamado utiliza uma transação que insere registros em duas tabelas simultaneamente — se qualquer etapa falha, o ROLLBACK garante que nenhum dado parcial persiste.

2.3 Controle de Concorrência e Recuperação de Falhas

Em sistemas multiusuário, múltiplas transações podem ser executadas simultaneamente, compartilhando o acesso aos mesmos recursos. Sem mecanismos de controle, problemas como leitura suja (dirty read), leitura não repetível (non-repeatable read) e atualização perdida (lost update) podem comprometer a integridade dos dados (ELMASRI; NAVATHE, 2011).

O controle de concorrência tem por objetivo garantir a propriedade de Isolamento das transações. O protocolo mais amplamente implementado nos SGBDs tradicionais é o bloqueio em duas fases (2PL — Two-Phase Locking), no qual uma transação adquire todos os bloqueios necessários antes de liberar qualquer um deles. SGBDs modernos como PostgreSQL utilizam o controle de concorrência por versão múltipla (MVCC — Multi-Version Concurrency Control), que permite leituras não bloqueantes ao manter versões históricas dos dados.

A recuperação de falhas é o mecanismo pelo qual o banco de dados restaura um estado consistente após falhas de transação, de sistema ou de mídia. A técnica mais utilizada é o arquivo de log de escrita antecipada (WAL — Write-Ahead Log), no qual todas as modificações são registradas em um log antes de serem aplicadas ao banco. Em caso de falha, as técnicas de undo (desfazer operações não confirmadas) e redo (refazer operações confirmadas) restauram o estado correto do banco (ELMASRI; NAVATHE, 2011).

No protótipo SmartTicket DB, o controle de concorrência é simulado por meio de threading.Lock, que garante exclusão mútua na seção crítica de atualização de chamados. A recuperação de falhas é representada pelo mecanismo de ROLLBACK automático em resposta a exceções.

2.4 Segurança, Integridade e DCL

A segurança em banco de dados abrange os mecanismos de controle de acesso que protegem os dados contra acessos não autorizados, modificações indevidas e divulgações inadequadas. O modelo de controle de acesso discricionário (DAC — Discretionary Access Control), implementado na DCL (Data Control Language), concede aos proprietários dos objetos a capacidade de determinar quem pode acessá-los e com que permissões.

Os principais comandos da DCL são GRANT, que concede privilégios específicos a usuários ou papéis (roles), e REVOKE, que os revoga. Em SGBDs como PostgreSQL e MySQL, é possível criar papéis com conjuntos de permissões predefinidos e atribuí-los a múltiplos usuários, simplificando a administração de segurança (ELMASRI; NAVATHE, 2011).

A integridade dos dados é garantida por restrições definidas no nível conceitual do banco: chaves primárias asseguram unicidade, chaves estrangeiras garantem a integridade referencial entre tabelas, restrições de domínio limitam os valores válidos para atributos, e restrições de entidade impedem valores nulos em atributos identificadores. O SQLite implementa estas restrições, mas requer a ativação explícita da verificação de chaves estrangeiras por meio do comando PRAGMA foreign_keys = ON.

2.5 Paradigmas Distribuídos: Teorema CAP e Replicação

Um banco de dados distribuído é um conjunto de bancos de dados logicamente inter-relacionados, distribuídos por uma rede de computadores, que o usuário percebe como um único sistema de banco de dados (ELMASRI; NAVATHE, 2011). A distribuição dos dados é realizada por meio de replicação (cópias dos dados em múltiplos nós) e fragmentação (divisão dos dados em partições distribuídas entre nós).

O Teorema CAP, enunciado por Brewer e formalizado por Gilbert e Lynch, estabelece que um sistema distribuído não pode garantir simultaneamente mais de duas das três seguintes propriedades: Consistência (C — todos os nós veem os mesmos dados no mesmo instante), Disponibilidade (A — toda requisição recebe uma resposta) e Tolerância a Partição (P — o sistema continua operando mesmo com falhas de comunicação entre nós) (MONIRUZZAMAN; HOSSAIN, 2013).

Uma extensão do Teorema CAP, denominada PACELC, introduzida por Abadi, estende a análise para situações sem partição de rede: mesmo em condições normais, há um trade-off entre latência (L) e consistência (C). Sistemas como o Cassandra e o DynamoDB favorecem disponibilidade e baixa latência, adotando um modelo de consistência eventual, enquanto sistemas como o Spanner do Google favorecem a consistência forte, aceitando maior latência.

No contexto do protótipo, a camada NoSQL de eventos documentais poderia, em ambiente de produção, operar em um cluster distribuído com replicação entre nós, adotando consistência eventual para suportar alto volume de eventos de log com baixa latência de escrita.

2.6 Bancos Móveis e Paradigma NoSQL

Os bancos de dados para aplicações móveis constituem uma extensão dos sistemas distribuídos adaptada às características peculiares dos dispositivos móveis: conectividade intermitente, energia limitada e requisitos de resposta em tempo real. O principal desafio técnico é a sincronização de dados entre o armazenamento local do dispositivo e o servidor central, com resolução de conflitos quando operações realizadas offline são submetidas ao servidor (GOOGLE, 2026).

O Firebase Firestore, da Google, implementa este modelo por meio de uma fila de operações pendentes armazenadas localmente. Quando a conectividade é restabelecida, o SDK sincroniza automaticamente as operações, resolvendo conflitos pela estratégia de last-write-wins ou por lógica customizada da aplicação. Este comportamento é simulado no SmartTicket DB pelos eventos NoSQL atualizacao_offline e sincronizacao_mobile.

Os bancos de dados NoSQL surgiram como resposta às limitações do modelo relacional frente ao crescimento exponencial dos dados na era da internet. Ao contrário dos SGBDs relacionais, que impõem um esquema rígido e garantem propriedades ACID completas, os sistemas NoSQL adotam o princípio BASE — Basically Available, Soft state, Eventual consistency — priorizando disponibilidade e escalabilidade horizontal sobre a consistência imediata (MONIRUZZAMAN; HOSSAIN, 2013).

Os quatro principais modelos NoSQL são: documental (MongoDB, Firestore), chave-valor (Redis, DynamoDB), orientado a colunas (Cassandra, HBase) e orientado a grafos (Neo4j). O modelo documental, adotado como referência neste trabalho, armazena os dados como documentos JSON ou BSON, com esquema flexível que permite a evolução da estrutura dos dados sem migrações de banco de dados (KHAN et al., 2023). MongoDB utiliza fragmentação (sharding) para distribuição horizontal dos dados, enquanto Firebase/Firestore oferecem sincronização em tempo real e suporte nativo a operações offline.

O MapReduce é um modelo de programação para processamento de grandes volumes de dados em clusters distribuídos, introduzido pelo Google. O paradigma divide o processamento em duas fases: a fase Map, na qual cada nó do cluster processa de forma independente uma partição dos dados, emitindo pares chave-valor intermediários; e a fase Reduce, na qual os pares intermediários com a mesma chave são agrupados e processados para produzir o resultado final. Embora frameworks como Hadoop e Spark implementem MapReduce em escala de petabytes, o conceito pode ser demonstrado localmente por operações simples de agrupamento e contagem sobre coleções de documentos.

3. Estudo de Caso: Arquitetura em Sistemas de Delivery e Atendimento

Os sistemas de delivery por aplicativo, como iFood e UberEats, constituem exemplos paradigmáticos da aplicação simultânea de todos os conceitos abordados neste trabalho. Sua arquitetura de dados é, por natureza, híbrida: combina um banco relacional transacional com bancos documentais de alta disponibilidade, além de sincronização mobile robusta para os dispositivos dos entregadores.

3.1 Escalabilidade, Segurança e Sincronização Off-line no Cenário Real

Camada SQL transacional. O processamento de pedidos e pagamentos exige propriedades ACID rigorosas. Uma transação de pedido insere o registro do pedido, desconta o saldo do usuário, registra a cobrança no gateway de pagamento e vincula um entregador disponível — tudo atomicamente. Uma falha em qualquer etapa deve acionar ROLLBACK para evitar cobranças sem entrega ou pedidos sem pagamento. O controle de acesso via DCL restringe quais sistemas podem modificar tabelas financeiras, utilizando papéis (roles) com permissões de INSERT e UPDATE apenas para os serviços autorizados.

Camada NoSQL documental. Milhões de eventos são gerados continuamente: atualizações de geolocalização dos entregadores a cada poucos segundos, mensagens entre usuários e estabelecimentos, logs de falhas e métricas de desempenho. O volume e a heterogeneidade desses dados inviabilizam o modelo relacional rígido. Bancos como Cassandra (otimizado para escritas de alta frequência) e Firestore (otimizado para sincronização em tempo real) são empregados para armazenar esses eventos documentais com escalabilidade horizontal (KHAN et al., 2023).

Concorrência. O problema de dois entregadores aceitando o mesmo pedido simultaneamente é a replicação exata do cenário de dois técnicos atualizando o mesmo chamado no SmartTicket DB. O SGBD utiliza bloqueios otimistas ou pessimistas para serializar o acesso e garantir que apenas um entregador seja vinculado a cada pedido.

Sincronização mobile. Entregadores frequentemente passam por áreas com sinal fraco ou sem cobertura. O aplicativo utiliza a persistência offline do Firebase/Firestore para armazenar atualizações de status localmente e sincronizá-las com o servidor central quando a conectividade é restabelecida. O protocolo de resolução de conflitos determina qual versão dos dados prevalece em caso de divergência entre o estado local e o estado do servidor (GOOGLE, 2026).

Relação com o protótipo. A lógica de "aceitar e resolver um chamado" no SmartTicket DB é estruturalmente idêntica à de "aceitar e concluir uma entrega": ambas envolvem uma transação SQL que atualiza o estado do objeto principal, registros de log documentais com eventos flexíveis, controle de concorrência para acesso simultâneo e sincronização de atualizações realizadas offline.

4. Protótipo Prático: SmartTicket DB

O SmartTicket DB é um protótipo acadêmico de sistema de gestão de chamados, implementado em Python com SQLite (camada SQL) e arquivo JSON local (camada NoSQL documental). Todas as bibliotecas utilizadas são nativas do Python 3, dispensando instalação de dependências externas. O protótipo é executado inteiramente via terminal com o comando python main.py.

4.1 Arquitetura Geral do Sistema

A arquitetura do SmartTicket DB é composta por duas camadas de persistência independentes, coordenadas por uma camada de serviço em Python. A Figura 1 ilustra os componentes do sistema e seus relacionamentos.

A camada SQL é responsável pelos dados estruturados e transacionais: perfis, usuários, chamados e logs relacionais. A camada de serviço orquestra as operações entre as duas camadas, aplicando o controle de acesso definido em auth.py antes de qualquer operação de escrita.

4.4 Demonstração Prática: ACID, Concorrência e Persistência Móvel

O arquivo main.py executa um roteiro completo de demonstração. Os principais cenários cobertos são descritos a seguir.

Atomicidade e ROLLBACK. A função abrir_chamado(simular_erro=True) lança uma exceção intencionalmente após inserir o chamado na tabela SQL, mas antes de inserir o log. O ROLLBACK desfaz a inserção do chamado, garantindo que nenhum registro parcial persiste no banco. A saída no terminal evidencia que o chamado não aparece na listagem final.

Consistência e integridade referencial. A tentativa de criar um chamado vinculado ao usuário de ID 9999 (inexistente) é rejeitada pelo SQLite com IntegrityError, demonstrando a restrição de chave estrangeira. O banco permanece em estado consistente.

Controle de acesso (DCL). O módulo auth.py define a matriz de permissões por perfil. A tentativa de um usuário com perfil cliente de executar a ação resolver_chamado é bloqueada pela função verificar_permissao e registrada como evento acesso_negado no banco NoSQL. A Figura 5 ilustra o fluxo de controle de acesso.

Concorrência. O módulo concurrency_demo.py lança duas threads representando dois técnicos que tentam atualizar o mesmo chamado simultaneamente. O threading.Lock serializa o acesso: a segunda thread aguarda a primeira concluir antes de prosseguir. O log de execução registra a ordem em que as atualizações ocorreram, evidenciando que nenhuma foi perdida.

Sincronização mobile. O sistema registra sequencialmente dois eventos NoSQL: atualizacao_offline (técnico registrou diagnóstico sem conexão) e sincronizacao_mobile (dados sincronizados com o servidor quando a rede foi restabelecida). Este comportamento simula diretamente o mecanismo de persistência offline do Firebase Firestore.

MapReduce conceitual. A função mapreduce_contar_por_tipo() lê todos os documentos da coleção JSON e conta as ocorrências de cada tipo de evento. A fase de mapeamento extrai o campo tipo de cada documento; a fase de redução soma as ocorrências por chave. O resultado é exibido em formato visual no terminal, com uma explicação da analogia ao paradigma MapReduce distribuído.

5. Resultados e Considerações Finais

O desenvolvimento do SmartTicket DB demonstrou, de forma prática e executável, a aplicabilidade dos conceitos fundamentais de banco de dados em um sistema funcional. Todos os requisitos do enunciado foram atendidos: a arquitetura ANSI/SPARC foi contextualizada na estrutura do protótipo; as propriedades ACID foram demonstradas com exemplos de COMMIT e ROLLBACK; o controle de concorrência foi simulado com threads e bloqueio explícito; a segurança e a integridade foram implementadas via perfis de acesso e chaves estrangeiras; e os conceitos de bancos distribuídos, móveis e NoSQL foram abordados tanto na fundamentação teórica quanto na implementação dos eventos documentais e da simulação de sincronização mobile.

A adoção de uma arquitetura híbrida SQL/NoSQL revelou-se uma escolha acertada para o contexto do sistema de chamados. A camada relacional garante a consistência dos dados transacionais, enquanto a camada documental oferece a flexibilidade necessária para o registro de eventos heterogêneos sem impor restrições de esquema. Esta combinação, observada em sistemas reais de alta escala como os aplicativos de delivery analisados, demonstra a complementaridade entre os dois paradigmas.

Como limitações do protótipo, destaca-se que o SQLite não implementa controle de acesso nativo com GRANT e REVOKE, como fazem PostgreSQL e MySQL, o que exigiu a simulação desta lógica na camada de aplicação. Adicionalmente, a implementação de persistência offline via JSON local é uma simplificação do comportamento real do Firebase Firestore, que inclui mecanismos sofisticados de resolução de conflitos e sincronização bidirecional.

Como desdobramentos futuros, o protótipo poderia ser expandido com a substituição do SQLite por PostgreSQL para demonstrar DCL nativa completa, a integração com MongoDB real por meio da biblioteca pymongo para persistência NoSQL distribuída genuína, e a implementação de uma interface de linha de comando interativa para exploração mais rica das funcionalidades. A adição de testes automatizados e a cobertura de cenários de recuperação de falhas com simulação de corrupção de dados também enriqueceriam a demonstração acadêmica.

Em síntese, o presente trabalho confirmou que os fundamentos de banco de dados — frequentemente apresentados de forma abstrata nos livros-texto — podem ser demonstrados de maneira concreta, executável e pedagogicamente eficaz por meio de um protótipo acessível, implementado com ferramentas nativas de uma das linguagens de programação mais amplamente adotadas no mercado.
